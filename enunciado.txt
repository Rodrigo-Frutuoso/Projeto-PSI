Projeto de Sistemas de Informação 
Enunciado do Projeto de 2025/2026 
1 Objetivo 
Pretende-se desenvolver uma aplicação web para gestão de coleções de álbuns 
de música em formato físico, que permita aos colecionadores organizar as suas 
coleções e consultar dados relevantes sobre os álbuns. A aplicação web deve seguir 
a arquitetura cliente-servidor, sendo sugerido o uso da MEAN stack:1 base de dados 
MongoDB, as frameworks Express e Angular para o desenvolvimento, respetiva
mente, do backend e frontend da aplicação, e o Node.js como ambiente de execu
ção de código JavaScript. 
O desenvolvimento da aplicação web é inspirado no modelo de processo de soft
ware Scrum, com três sprints de duas semanas cada.2 Em cada sprint pretende-se 
que seja implementada uma parte das user stories da aplicação web, havendo uma 
demonstração logo após o final de cada sprint. 
Neste documento é inicialmente apresentado o universo do discurso bem como as 
user stories a implementar. No final do documento, são descritos requisitos adici
onais e recomendações, assim como detalhes relativos às entregas de relatórios e 
avaliações do projeto.

3 User stories 
A seguir são enumeradas e descritas as user stories que a aplicação web deve su
portar, cuja implementação deverá decorrer ao longo de três sprints. 
3.1 Sprint 1 
User Story 1: Como utilizador da aplicação, quero poder criar uma conta de utili
zador, para ter acesso à mesma. Os critérios de aceitação são os seguintes: 
a) Para criar uma conta de utilizador, o utilizador deve fornecer um nome de uti
lizador, o seu e-mail, palavra-passe, e data de nascimento. 
b) O nome de utilizador e o e-mail devem ser diferentes dos de quaisquer outros 
utilizadores. 
c) O nome de utilizador só pode ter letras e dígitos. 
d) A palavra-passe deve ter oito ou mais carateres, incluindo pelo menos uma 
letra maiúscula, uma letra minúscula, e um algarismo. 
e) O utilizador tem de ter pelo menos 13 anos de idade. 
f) Os requisitos de construção do nome de utilizador e palavra-passe devem ser 
apresentados no ecrã de criação de conta. 
g) Em caso de sucesso, uma mensagem adequada deve ser mostrada. 
h) Se o nome de utilizador, palavra-passe, e/ou idade não cumprirem os requisi
tos, uma mensagem de erro deve indicar qual ou quais os requisitos que não 
foram cumpridos. 
User Story 2: Como utilizador da aplicação, quero poder iniciar sessão na aplica
ção, para depois poder aceder às suas funcionalidades. Os critérios de aceitação 
são os seguintes: 
a) Para iniciar a sessão, o utilizador deve usar o nome de utilizador e a palavra
passe que definiu aquando da criação da sua conta.

4 Requisitos adicionais 
As tarefas de implementação das user stories da aplicação web devem ter em conta 
os requisitos adicionais indicados nesta secção. 
Responsive web design. A aplicação deve adaptar a sua interface consoante o tipo 
de dispositivo do utilizador, nomeadamente ecrãs de diferentes tamanhos. 
Segurança. As palavras-passe dos utilizadores devem ser cifradas antes de serem 
guardadas na base de dados. Adicionalmente, deve ser iniciada uma sessão auten
ticada antes de serem realizados pedidos ao frontend e backend da aplicação que 
envolvam dados do utilizador, com exceção da criação da conta do utilizador. 
Dados de exemplo. A base de dados da aplicação deve ser inicializada com um 
número razoável de artistas, álbuns, e versões, que permita a demonstração expe
dita das funcionalidades implementadas em cada sprint. O script ou rota de inicia
lização deve ser providenciado juntamente com o código entregue, para facilitar a 
execução do projeto por parte dos docentes. 
Imagens. Por limitações nas atividades de submissão de trabalhos do Moodle (cujo 
limite é 100 MBytes), as imagens da aplicação web, incluindo as capas de álbuns, 
não devem exceder o limite de 200 KBytes cada. Alternativamente, as imagens po
dem ser alojadas externamente, desde que fique assegurado que as mesmas são 
apresentadas corretamente durante a execução do projeto pelos docentes. 
Gestão do projeto. As tarefas do projeto e a sua execução devem ser registadas 
usando a ferramenta Planner, abordada numa aula teórico-prática, disponível no 
Moodle. Para o efeito, cada grupo dispõe de um plano de projeto vazio, com nome 
PLANO-PSI-XX (onde XX é o número do grupo com dois dígitos, por exemplo 01), no 
qual as pessoas do grupo podem contribuir para a gestão e acompanhamento do 
projeto. O plano será mostrado aquando de cada demonstração da aplicação. 
5 Recomendações 
Seguem-se algumas recomendações sobre o trabalho a realizar. 
Inicialização de dados. Recomenda-se a criação de uma rota no backend da apli
cação que facilite a inicialização da base de dados (por exemplo, /api/init). 
Design system. Recomenda-se a reutilização de um design system existente, para 
desenvolver mais rapidamente uma interface apelativa, coerente, e familiar ao uti
lizador. Numa aula teórica, disponível no Moodle, o tema de design systems foi 
9 
abordado, tendo sido mostrados vários exemplos, acompanhados de hiperligações 
para descarregar recursos e obter mais informações. 
Desenho da API. A Application Programming Interface do backend da aplicação 
deve ser desenhada considerando boas práticas da indústria,13 incluindo o formato 
dos dados dos pedidos e respostas, os nomes das rotas, os códigos de estado HTTP, 
e outros aspetos. 
Controlo de versões. Recomenda-se vivamente a utilização de um repositório git 
para a gestão colaborativa de versões do projeto, evitando percalços como a perda 
acidental do código. Pode ser criado um repositório no git do DI/FCUL,14 ou, se pre
ferível, no Github,15 Bitbucket,16 ou GitLab.17 
6 Entregas 
No final de cada sprint, cada grupo deve preparar um ficheiro ZIP com o seguinte 
conteúdo: a) o código das user stories implementadas durante o sprint, abrangendo 
o frontend e o backend; b) um diagrama da estrutura da base de dados, gerado por 
uma aplicação como o Compass18 ou o DBeaver;19 e c) um ficheiro com instruções, 
ou dockerfile, contendo os comandos necessários à execução da aplicação.

e deve ter o nome PSI_Sx_TPyy_Gzz.ZIP, onde x é o número do sprint, yy é o número da 
turma teórico-prática do grupo, e zz é o número do grupo (com dois dígitos, por 
exemplo, 01).

7 Avaliações 
O projeto é avaliado com base no cumprimento dos critérios de aceitação das user 
stories dos sprints, tendo todos os sprints o mesmo peso, bem como na discussão 
f
inal com o grupo. A nota no projeto é atribuída individualmente. 
Cada sprint é avaliado em laboratório na aula teórico-prática da turma em que o 
grupo está inscrito, nas seguintes semanas: 
• Sprint 1: de 27 a 30 de abril;21 
• Sprint 2: de 11 a 15 de maio; 
• Sprint 3: de 25 a 29 de maio e de 1 a 3 de junho. 
É obrigatória a presença de todas as pessoas do grupo durante as avaliações. 
A avaliação de cada sprint consiste na demonstração das funcionalidades imple
mentadas, podendo ser realizada num computador portátil e devendo ser coerente 
com a versão do código entregue. 
No último sprint, para além da demonstração da aplicação web, tem lugar a discus
são final com o grupo, razão pela qual as avaliações acontecem também nas aulas 
teóricas, estendendo-se por duas semanas.
